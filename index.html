1敏捷核心考点

产品负责人 PO 的职责

  • PO 是客户、业务与开发团队之间的桥梁
  • 需求不清、用户故事不明确时，应召集团队与 PO 协作澄清
  • PO 关注：产品价值、需求优先级、客户反馈，而非主要关注成本
  • 产品需求不明确时 → 先明确产品愿景，统一方向，再细化需求

产品待办事项列表 Product Backlog

  • 动态文档，记录、跟踪和管理产品需求及变更
  • 项目暂停后恢复 → 先回顾剩余待办，重新确认优先级和环境
  • 待办事项梳理会议：帮助跨地域、跨团队同步用户故事和需求变更

用户故事与接受标准

  • 用户故事需足够细节，支持团队开发所需功能
  • 故事过大、估算不准 → 拆分为更小故事再估算
  • 价值判断应与业务目标、项目目标和战略目标对齐

完成的定义 DoD

  • 统一团队与干系人对"完成"的理解
  • 作用：保证故事完整性、提高交付质量、避免期望落差、确保合规
  • 客户认为成果未达预期 → 往往是 DoD 或验收标准未定义好

信息发射源

  • 看板、燃尽图、任务墙、仪表盘等
  • 作用：可视化项目状态、展示进度/风险/问题、明确范围变更
  • 敏捷强调：透明、及时、可视化沟通

敏捷团队与仆人式领导

  • 团队应自组织、自管理，对工作进度负责
  • 及时更新看板，保证信息反映真实状态
  • 项目经理作为仆人式领导应：
    • 授权团队、移除障碍、促进讨论
    • 提醒团队责任、避免微观管理
    • 营造安全、信任的环境

MVP 与增量交付

  • 第一次迭代优先交付 MVP（最小可行产品）
  • MVP 用最小成本展示核心价值，尽早获得反馈
  • 增量交付适合：需快速上市、保留核心功能按期交付、后续逐步增加功能
  • 动态范围合同：固定时间和预算内，通过迭代交付最高价值范围

敏捷质量与绩效

  • 可交付成果缺陷数量可反映交付质量
  • 单位工作量错误更少的团队表现更好
  • 冲刺评审关注产品增量和团队成果，不评价个人表现

2沟通与干系人管理

干系人识别与分析

  • 新发现干系人时，应更新：
    • 干系人登记册
    • 干系人参与计划
  • 干系人登记册记录：角色、权力、利益、影响、期望等
  • 干系人分析：识别权力、利益、影响力和期望，制定沟通策略
  • 高影响力干系人提出变更 → 不能直接处理，先完善干系人识别和管理基础

干系人期望管理

  • 干系人需求不一致 → 通过分析、征求意见、谈判平衡需求
  • 用户角色 Persona：帮助理解不同用户群体的需求和期望
  • 干系人质疑项目价值 → 说明项目使命、愿景、目标、预期成果及战略关系
  • 高层关注战略价值，强调项目目标与组织战略目标一致

沟通管理计划

  • 规定：沟通对象、方式、频率、内容、发布渠道、是否确认收到
  • 没收到报告、供应商未回复、信息未传达 → 通常是沟通管理计划问题
  • 向供应商发送关键信息时，应确认对方已收到

沟通方式

  • 互动沟通：会议、电话、视频、即时通讯 → 实时多方交流
  • 推式沟通：邮件、报告、备忘录 → 能发送但不能保证理解
  • 拉式沟通：门户网站、知识库、内网 → 大量复杂信息或大量受众
  • 分散团队需要统计反馈 → 调查问卷优于邮件和会议

跨地域与跨文化团队沟通

  • 跨时区协作 → 合理规划任务，减少跨时区依赖
  • 共址但工作时间不同 → 寻找共同重叠时间安排会议
  • 跨文化沟通障碍 → 建立共同目标和共同理解
  • 跨国敏捷团队 → 需求或接受标准变更应通过待办事项梳理同步所有团队

3范围、需求与变更管理

启动阶段需求

  • 启动阶段明确高层次业务需求，为项目章程和后续规划奠定基础
  • 需求不明确 → 先澄清高级别需求和产品待办列表
  • 业务需求应描述项目要解决的核心问题和预期价值

范围说明书

  • 定义项目边界、可交付成果、验收标准、制约因素、假设条件
  • 项目范围与产品范围不同：前者是项目产生的工作，后者是产品/服务的特性
  • 范围说明书是整体变更控制的基准

范围蔓延与镀金

  • 范围蔓延：未经批准添加额外工作 → 拒绝，立即进入整体变更控制
  • 镀金：团队主动做范围外工作讨好客户 → 禁止，会导致进度/成本问题
  • 客户额外变更 → 必须实施整体变更控制

整体变更控制

  • 所有变更必须书面记录、评估影响、获得批准
  • 变更日志跟踪变更请求的状态和历史
  • 批准或拒绝变更时需考虑：对范围、进度、成本、质量的影响

敏捷中的变更

  • 敏捷欢迎变更，即使在开发后期也欢迎
  • 迭代评审会上展示已完成功能，收集反馈调整待办事项
  • 未在当前迭代中实现的变更 → 移入产品待办列表重新排优先级

4进度与成本管理

关键路径法

  • 关键路径是项目中最长的路径，决定最短完成时间
  • 关键路径上的活动没有浮动时间
  • 压缩工期选项：
    • 赶工：加资源、加成本（适用于关键路径活动）
    • 快速跟进：并行执行有依赖关系的活动、加风险

三点估算

  • 适用于估算不确定性较大的活动
  • 公式：（乐观 + 4×最可能 + 悲观）/ 6
  • 乐观(O)、最可能(M)、悲观(P) 三点估计

挣值管理 EVM

  • CV（成本偏差）= EV - AC；CV > 0 优于预算
  • SV（进度偏差）= EV - PV；SV > 0 优于进度
  • CPI（成本绩效指数）= EV / AC；< 1 表示超支
  • SPI（进度绩效指数）= EV / PV；< 1 表示延误
  • ETC = EAC - AC（完工尚需估算）

成本超支处理

  • 展示绩效数据 → 提出纠正措施 → 预防未来超支
  • 分析原因：范围蔓延、资源效率低、估算不准
  • 考虑：重新规划、调整范围、申请额外资源

5风险与质量管理

风险识别与分析

  • 风险是一种不确定的事件或情况，可能发生也可能不发生
  • 已发生的问题 → 问题日志，不再是风险
  • 风险分析：概率影响矩阵确定风险优先级

蒙特卡洛分析

  • 模拟大量随机场景，计算完成概率、超支概率
  • 回答"我们有多少把握按时完成？"

决策树分析

  • 适用于多方案选择，考虑概率和收益
  • 计算每个方案的预期货币价值 EMV
  • EMV = Σ（概率 × 收益）

质量问题处理

  • 可交付成果不合格 → 更新问题日志，实施解决方案
  • 先分析原因，再采取纠正措施
  • 根本原因分析：鱼骨图、5 Why、帕累托图

质量保证 vs 质量控制

  • 质量保证：过程审查，确保遵循过程 → 预防缺陷
  • 质量控制：成果检查，识别缺陷 → 发现问题

6团队建设与冲突管理

塔克曼阶梯

  • 形成期：团队初次见面，了解目标和角色
  • 震荡期：冲突显现，需要领导引导
  • 规范期：建立信任，协作顺畅
  • 成熟期：高绩效，自我管理
  • 项目结束/人员变动可能回到早期阶段

冲突管理策略

  • 撤退/回避：暂时离开，让双方冷静（不是最佳）
  • 缓和/包容：强调一致，忽略差异
  • 妥协/调解：双方各让一步
  • 强迫/命令：一方赢，一方输
  • 合作/解决问题：最佳方案，双方都满意

团队章程与规则

  • 核心原则：对事不对人、聚焦解决方案、尊重沟通、明确行为准则
  • 应提前建立互动规则

新成员加入

  • 需要紧密协作时 → 组织全体团队研讨会
  • 目的：知识转移、建立关系、对齐目标、营造协作环境

激励团队

  • 采取激励措施前 → 先了解团队成员的需求、愿望和动机
  • 跨国团队破冰 → 通过团队自我评估引导成员表达期望

7采购与项目收尾

采购流程

  • 选择新供应商 → 遵循组织和项目规定的正式采购流程
  • 与供应商谈判前 → 了解合同政策、法律和公司标准

采购管理计划

  • 规定采购过程中的活动、角色和可交付成果
  • 相关方对采购可交付成果有不同意见 → 参考采购管理计划
  • 发起人询问为何内部开发 → 查看采购管理计划中的自制或外购决策

供应商沟通

  • 不能只发送邮件 → 还应确认对方收到
  • 供应商未回复导致资源不到位 → 本质是沟通确认机制不足

收尾阶段核心工作

  • 验收成果 → 移交成果 → 关闭合同 → 归档文件
  • 总结经验教训 → 更新组织过程资产 → 释放资源

移交运营

  • 为降低风险 → 提前部署测试版本，让运营团队参与早期发布
  • 让运营团队提前熟悉产品，降低正式交接风险

项目结束后的费用

  • 项目已完成并移交后，客户提出的运营或团队建设费用不属于项目范围
  • 应急储备用于项目内已识别风险，不应用于项目结束后的运营

组织过程资产 OPA

  • 包括：流程、程序、模板、经验教训、知识库、历史信息
  • 项目收尾时应更新 OPA
  • 人员流动频繁时 → 更新 OPA 有助于保持治理连续性

★压缩记忆版

📌 10 条核心记忆

  1. 敏捷重价值：PO 管需求，团队自组织，PM 做仆人式领导
  2. 需求不清先澄清：产品愿景、用户故事、接受标准、DoD
  3. 透明靠信息发射源：看板、燃尽图、仪表盘
  4. 变更走流程：客户额外变更必须实施整体变更控制
  5. 干系人先识别再管理：登记册、分析、参与计划、沟通策略
  6. 沟通问题看沟通管理计划：未收到、未确认、信息不同步
  7. 进度压缩两招：赶工加资源加成本，快速跟进并行加风险
  8. 风险未发生，问题已发生：风险用矩阵/蒙特卡洛/决策树；问题进问题日志
  9. 质量问题先分析原因：不合格更新问题日志，做根本原因分析
  10. 收尾要验收、移交、归档、总结、释放资源
